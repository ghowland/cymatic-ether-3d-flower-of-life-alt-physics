# Language Server Protocol as Universal Development Environment for Silo

**Author:** Claude (Anthropic)  
**Date:** January 29, 2026  
**Context:** Technical analysis of LSP integration with Silo's MemDB architecture

---

## Abstract

This paper examines how Language Server Protocol (LSP) transforms from a code intelligence tool into a complete development environment when applied to Silo's data-centric architecture. By implementing an LSP server that treats MemDB as its source of truth, Silo provides developers with code editor ergonomics while maintaining strict data validation, live debugging, interactive querying, and collaborative editing—all through the text editor interface developers already know. The result is a development environment where the boundary between "writing code" and "querying/modifying live data" disappears entirely.

---

## 1. Introduction: The LSP Opportunity

### 1.1 What LSP Provides

Language Server Protocol separates language intelligence from editor implementation:

**Traditional model:**
```
Each editor implements:
- Syntax highlighting
- Autocomplete
- Error checking
- Go to definition
- Find references
× 20 editors = 20 implementations
```

**LSP model:**
```
One LSP server provides:
- All language intelligence
- All validation
- All queries

Works with 20+ editors automatically
```

### 1.2 Why Silo Is Perfect For LSP

**Traditional LSP servers analyze:**
- Static source files
- File system structures
- Import/export declarations
- Type definitions in code

**Silo's LSP server analyzes:**
- Live MemDB state
- Runtime entity data
- Active prolog facts
- Current game state

**The difference:**

Traditional LSP: "What could this code do?"  
Silo LSP: "What is this data doing right now?"

## 2. Core Capabilities

### 2.1 Validation as Database Constraint Checking

**Traditional code validation:**
```typescript
// LSP checks syntax and types
function attack(target: Enemy) {
    target.health -= 10;  // Type checker: ✓
}
```

**Silo validation:**
```prolog
// LSP checks against MemDB schema
can_attack(Entity, Target) :-
    health(Entity, H), H > 50,  // LSP: Entity must be valid ID
    distance(Entity, Target, D), D < 100.  // LSP: Target must exist
```

**LSP validation queries:**
1. Does `Entity` reference a valid entity ID in MemDB?
2. Does `health` exist as a queryable stat?
3. Are comparison operators valid for stat types?
4. Does `distance` function exist in Prolog runtime?

**Not syntax checking—schema validation.**

### 2.2 Autocomplete from Live Data

**Traditional autocomplete:**
```javascript
// Shows: All possible properties of Enemy class
enemy.|
      ├─ health
      ├─ position
      ├─ attack()
      └─ [100 more members]
```

**Silo autocomplete:**
```prolog
// Type: health(
// LSP queries MemDB and shows:
health(Entity, Value)

Available entities (live):
  - Skeleton_5 (health: 35)
  - Player_1 (health: 100)
  - Goblin_12 (health: 22)

Available stats:
  - Health
  - Stamina
  - Mana
```

**Implementation:**
```zig
fn handleAutocomplete(cursor_position: Position, context: Text) []CompletionItem {
    // Parse partial text
    const partial = parsePartialQuery(context);
    
    // Query MemDB for valid completions
    switch (partial.type) {
        .entity_id => return db.getAllEntityIds(),
        .stat_name => return db.getAllStatTypes(),
        .skill_name => return db.getAllSkillTypes(),
    }
}
```

**Completions are queries, not static lists.**

### 2.3 Error Checking Against Runtime State

**Text editor shows:**
```prolog
can_attack(Entity_999, Target) :-  // Red squiggle under Entity_999
    health(Entity_999, H), H > 50.
```

**LSP error:**
```
Entity_999 does not exist in game.entity table.
Did you mean: Entity_5, Entity_12, Entity_27?
```

**This is impossible in traditional LSP** because the language server doesn't have access to runtime state. Silo's LSP does because MemDB is both compile-time schema and runtime data.

const Watcher = struct {
    path: Text,
    last_value: BlockValue,
    client_id: i32,
};

var active_watchers: []Watcher = &[_]Watcher{};

fn updateWatchers() void {
    for (active_watchers) |*watcher| {
        const current = dr.getAny(watcher.path);
        if (!equal(current, watcher.last_value)) {
            sendLSPNotification(watcher.client_id, current);
            watcher.last_value = current;
        }
    }

## 5. Technical Implementation

### 5.1 LSP Server Architecture

```zig
pub const SiloLSPServer = struct {
    // Connection to MemDB
    db: *MemDB,
    
    // Active watchers for live updates
    watchers: []Watcher,
    
    // Prolog evaluator for queries
    prolog: *PrologEngine,
    
    // LSP protocol handlers
    text_document_handlers: TextDocumentHandlers,
    workspace_handlers: WorkspaceHandlers,
    
    pub fn handleRequest(
        self: *SiloLSPServer,
        method: Text,
        params: json.Value
    ) json.Value {
        if (method.equals("textDocument/completion")) {
            return self.handleCompletion(params);
        }
        if (method.equals("textDocument/hover")) {
            return self.handleHover(params);
        }
        if (method.equals("silo/query")) {
            return self.handleQuery(params);
        }
        if (method.equals("silo/watch")) {
            return self.handleWatch(params);
        }
    }
};
```

### 5.2 The Query Handler

```zig
fn handleQuery(self: *SiloLSPServer, params: json.Value) json.Value {
    const query_text = params.get("query").asText();
    
    // Parse Prolog query
    const query = self.prolog.parse(query_text) catch |err| {
        return jsonError(err);
    };
    
    // Execute against MemDB fact sets
    const results = self.prolog.query(
        self.db.fact_sets,
        query
    );
    
    // Format results for LSP
    var response = json.Array.init();
    for (results) |result| {
        response.append(formatBinding(result));
    }
    
    return json.Object{
        .results = response,
        .count = results.len,
        .execution_time_ms = getTimestamp() - start,
    };
}
```

### 5.3 The Autocomplete Handler

```zig
fn handleCompletion(
    self: *SiloLSPServer,
    params: json.Value
) json.Value {
    const position = params.get("position");
    const text = params.get("textDocument").get("text");
    
    // Parse context around cursor
    const context = parseContext(text, position);
    
    var completions = json.Array.init();
    
    // Determine what we're completing
    switch (context.type) {
        .entity_reference => {
            // Query all entity IDs from MemDB
            const entities = self.db.getAllEntities();
            for (entities) |entity| {
                completions.append(.{
                    .label = formatEntityId(entity.id),
                    .kind = .Variable,
                    .detail = formatEntityType(entity.entity_type),
                    .documentation = formatEntityState(entity),
                });
            }
        },
        .stat_name => {
            // Complete with stat types
            for (StatType.values()) |stat| {
                completions.append(.{
                    .label = @tagName(stat),
                    .kind = .Property,
                });
            }
        },
        .prolog_predicate => {
            // Complete with available predicates
            const predicates = self.prolog.getAvailablePredicates();
            for (predicates) |pred| {
                completions.append(.{
                    .label = pred.name,
                    .kind = .Function,
                    .detail = pred.signature,
                });
            }
        },
    }
    
    return .{ .items = completions };
}
```

### 5.4 The Watcher System

```zig
fn handleWatch(self: *SiloLSPServer, params: json.Value) json.Value {
    const path = params.get("path").asText();
    const client_id = params.get("clientId").asI32();
    
    // Add watcher
    const watcher = Watcher{
        .path = path,
        .last_value = dr.getAny(path),
        .client_id = client_id,
    };
    
    self.watchers.append(watcher);
    
    return .{ .success = true };
}

// Called every frame
pub fn updateWatchers(self: *SiloLSPServer) void {
    for (self.watchers) |*watcher| {
        const current = dr.getAny(watcher.path);
        
        if (!valuesEqual(current, watcher.last_value)) {
            // Send notification to client
            self.sendNotification(watcher.client_id, .{
                .method = "silo/watchUpdate",
                .params = .{
                    .path = watcher.path,
                    .old_value = watcher.last_value,
                    .new_value = current,
                    .timestamp = getTimestamp(),
                },
            });
            
            watcher.last_value = current;
        }
    }
}
```

### 5.5 The Validation Handler

```zig
fn handleDiagnostics(
    self: *SiloLSPServer,
    document: TextDocument
) []Diagnostic {
    var diagnostics = ArrayList(Diagnostic).init();
    
    // Parse document as Prolog
    const rules = self.prolog.parseAll(document.text) catch |err| {
        diagnostics.append(.{
            .range = getErrorRange(err),
            .severity = .Error,
            .message = formatError(err),
        });
        return diagnostics.items;
    };
    
    // Validate each rule against MemDB schema
    for (rules) |rule| {
        for (rule.conditions) |condition| {
            // Check if referenced entities exist
            if (condition.type == .entity_reference) {
                const id = condition.entity_id;
                if (!self.db.entityExists(id)) {
                    diagnostics.append(.{
                        .range = condition.range,
                        .severity = .Error,
                        .message = formatStr(
                            "Entity {} does not exist",
                            .{id}
                        ),
                    });
                }
            }
            
            // Check if stats are valid
            if (condition.type == .stat_query) {
                if (!self.db.isValidStatType(condition.stat_name)) {
                    diagnostics.append(.{
                        .range = condition.range,
                        .severity = .Error,
                        .message = formatStr(
                            "Unknown stat type: {}",
                            .{condition.stat_name}
                        ),
                    });
                }
            }
        }
    }
    
    return diagnostics.items;
}
```

## 8. Performance Considerations

### 8.1 Query Optimization

**Problem:** Querying entire MemDB every autocomplete is expensive.

**Solution:** Incremental indexing.

```zig
pub const LSPIndex = struct {
    entity_ids: HashMap(i32, EntityInfo),
    stat_types: []StatType,
    skill_types: []SkillType,
    
    last_update_frame: i32,
    dirty: bool,
    
    pub fn update(self: *LSPIndex, db: *MemDB) void {
        if (db.frame_number == self.last_update_frame) return;
        
        // Only reindex if data changed
        if (db.entities_modified) {
            self.reindexEntities(db);
        }
        
        self.last_update_frame = db.frame_number;
        self.dirty = false;
    }
};
```

**Reindex only when MemDB changes, cache results.**

### 8.2 Watcher Throttling

**Problem:** Streaming every change at 60fps overwhelms network.

**Solution:** Debouncing and rate limiting.

```zig
pub const Watcher = struct {
    path: Text,
    last_value: BlockValue,
    last_sent_value: BlockValue,
    last_sent_time: f32,
    update_interval: f32 = 0.1,  // Max 10 updates/sec
    
    pub fn shouldSend(self: *Watcher, current_time: f32) bool {
        // Changed and enough time passed?
        return !valuesEqual(self.last_value, self.last_sent_value) and
               (current_time - self.last_sent_time) > self.update_interval;
    }
};
```

### 8.3 Lazy Loading

**Don't load all 100MM records into LSP server.**

```zig
// Only load current game's data on startup
pub fn initialize(game_id: i32) void {
    loadEntities(game_id);
    loadSkills(game_id);
    loadStateMachines(game_id);
    // Other games' data loaded on-demand
}

// Load other games lazily when referenced
pub fn resolveReference(entity_id: i32) EntityInfo {
    if (!isLoaded(entity_id)) {
        loadEntityFromDB(entity_id);
    }
    return getCached(entity_id);
}
```

## 9. Security and Isolation

### 9.1 Sandboxing Queries

**Problem:** Queries could be expensive or malicious.

**Solution:** Resource limits and timeouts.

```zig
pub fn executeQuery(query: Text) !QueryResult {
    const sandbox = Sandbox{
        .max_execution_time_ms = 100,
        .max_results = 1000,
        .max_memory_mb = 10,
    };
    
    return sandbox.run(|| {
        return prolog.query(fact_sets, query);
    });
}
```

### 9.2 Multi-User Isolation

**Problem:** Multiple developers shouldn't interfere.

**Solution:** Separate MemDB instances per user, with optional sharing.

```zig
pub const LSPServer = struct {
    instances: HashMap(UserId, MemDBInstance),
    
    pub fn handleRequest(user_id: UserId, request: Request) Response {
        const instance = self.instances.get(user_id);
        return instance.handle(request);
    }
};
```
